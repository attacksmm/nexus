Архитектура модуля (ZIP)

Модуль — это ZIP-архив со строгой структурой. Минимальный набор: manifest.json + router.py.

my-module.zip
├── manifest.json             # ОБЯЗАТЕЛЬНО — метаданные модуля
├── router.py                 # ОБЯЗАТЕЛЬНО — FastAPI APIRouter
├── panel/
│   ├── index.html            # ОБЯЗАТЕЛЬНО — UI панели управления (iframe)
│   └── docs.html             # РЕКОМЕНДУЕТСЯ — документация модуля (кнопка ?)
└── static/                   # Опционально — CSS/JS/изображения

manifest.json — обязательные поля

{
  "id": "my-module",           // Уникальный ID: только a-z, 0-9, дефис. Используется в URL.
  "name": "Мой модуль",        // Отображаемое имя в sidebar
  "version": "1.0.0",          // Семантическая версия
  "description": "..."         // Краткое описание (отображается в UI)
}

Загружается в <iframe>. URL: /nexus/{id}/panel/index.html.

Для запросов к API модуля используйте относительный путь, вычисляя базу из URL страницы:

const parts = location.pathname.split('/');
const panelIdx = parts.indexOf('panel');

panel/docs.html — документация модуля

Открывается кнопкой ? в toolbar. Должна содержать:

  • Назначение модуля
  • Полный список API-эндпоинтов (метод, путь, параметры, ответ)
  • Структуру БД (таблицы, поля, типы)
  • Примеры запросов
  • Настройки и зависимости

# 2. manifest.json — id должен быть уникальным
# 3. router.py — APIRouter + setup(ctx)
# 4. panel/index.html — UI (самодостаточный HTML)

# 7. Загрузить через UI (кнопка + Модуль) или curl:

Требования к docs.html

Файл panel/docs.html должен быть написан так, чтобы его было достаточно для полного понимания модуля — в том числе ИИ-ассистентом. Обязательные разделы:

  1. Назначение — что делает модуль, для кого, в каком контексте
  2. Принцип работы — архитектура, поток данных
  3. Структура БД — все таблицы, поля, типы, индексы
  4. REST API — метод, путь, параметры, пример запроса, пример ответа
  5. Зависимости — внешние библиотеки, API, сервисы
  6. Настройки — конфигурационные параметры и как их получить

Технические ограничения

  • Модуль монтируется динамически через importlib — нельзя использовать __file__ для путей; используйте ctx.module_dir
  • Все async операции в setup() должны запускаться через asyncio.get_event_loop().create_task()
  • Модуль получает изолированную SQLite БД в ctx.db_path
  • Логи модуля: ctx.logger → файл data/logs/module.log (ротация 5MB×3)
  • При выгрузке удаляется вся папка modules/{id}/ включая данные
