База клиентов v2.1.3

Модуль для хранения клиентов в SQLite. Поддерживает несколько изолированных таблиц. Каждая запись содержит три поля: id, platform_id, custom_fields.

Структура записи

ПолеТипОписание
idINTEGERАвтоинкремент, уникальный номер строки
platform_idTEXTID клиента на внешней платформе (ВКонтакте, Telegram, CRM и т.д.)
custom_fieldsJSONПроизвольные поля: имя, телефон, email, теги — любые ключ-значение
created_atTEXTISO-дата создания
updated_atTEXTISO-дата последнего изменения

Таблицы

Один модуль может содержать несколько таблиц (например, отдельно для каждой платформы). Таблица default создаётся автоматически и не удаляется. В панели она отображается как Основная и работает как сводный лист: показывает и ищет строки из всех таблиц сразу, сохраняя признак исходного листа.

REST API

Базовый путь: /nexus/customer-db/api

Таблицы

МетодПутьОписание
GET/tablesСписок всех таблиц
POST/tablesСоздать таблицу {name, display_name, description}
DELETE/tables/{name}Удалить таблицу (кроме default)
GET/statsКоличество записей по всем таблицам

Записи

МетодПутьОписание
GET/tables/{table}/recordsСписок записей. Параметры: q, limit, offset
POST/tables/{table}/recordsСоздать запись {platform_id, custom_fields}
POST/tables/{table}/records/upsertСоздать или обновить одну запись по platform_id. Удаляет пустые значения и неподставленные плейсхолдеры #{...}.
POST/tables/{table}/records/batch-upsertПачками до 1000 записей: создать новые и обновить существующие по platform_id
POST/{table}Webhook-friendly upsert: endpoint называется как таблица, например /vk_clients. Принимает nested JSON, form-data и плоские ключи custom_fields.name.
GET/tables/{table}/records/{id}Одна запись по id
PUT/tables/{table}/records/{id}Обновить запись
DELETE/tables/{table}/records/{id}Удалить запись
GET/tables/{table}/statsИтого и сегодня

Хранение и обслуживание

МетодПутьОписание
GET/maintenance/storageРазмер диска, live DB, archive DB, backup-кандидаты и топ таблиц по размеру
GET/maintenance/archive/plan?months=6План переноса старых строк в локальный архив без изменения данных
POST/maintenance/archive/run?months=6&dry_run=trueDry-run архивации. При dry_run=false переносит старые строки в archive DB и удаляет их из live DB только после записи в архив
GET/maintenance/compact/plan?tables=vk_clients,telegram_clientsОценка экономии от сжатия тяжёлых custom_fields
POST/maintenance/compact/run?dry_run=trueDry-run compaction. При dry_run=false полный JSON сохраняется gzip в payload-хранилище, а в live DB остаётся компактный профиль
GET/maintenance/backups/plan?keep_latest=2Показывает, какие customer-db backup-копии будут сохранены и удалены
POST/maintenance/backups/cleanup?keep_latest=2&execute=falseDry-run очистки. Удаление выполняется только при execute=true

Архив и payload-хранилище лежат локально в modules/customer-db/data/archive/customer-db-archive.db. Списки, статистика и карточки читают live + archive прозрачно; архивные записи получают служебный признак archived: true, но основные поля ответа остаются прежними. При compaction полный custom_fields не удаляется: он автоматически восстанавливается при чтении записи/API.

Пример запроса

POST /nexus/customer-db/api/tables/default/records
Content-Type: application/json

{
  "platform_id": "vk_123456",
  "custom_fields": {
    "name": "Иван Иванов",
    "phone": "+7 999 000 11 22",
    "email": "ivan@example.com",
    "source": "ВКонтакте"
  }
}

Одиночный upsert для ботов

POST /nexus/customer-db/api/tables/telegram_clients/records/upsert
Content-Type: application/json

{
  "platform_id": "123456",
  "custom_fields": {
    "platform": "telegram",
    "salebot_id": "123",
    "utms": {
      "utm_campaign": "spring"
    }
  }
}

Webhook endpoint по имени таблицы

POST /nexus/customer-db/api/vk_clients
Content-Type: application/json

{
  "platform_id": "123456",
  "custom_fields": {
    "name": "Иван",
    "second_name": "Иванов",
    "phone": "+7 999 000 11 22",
    "email": "ivan@example.com",
    "utm_source": "senler"
  }
}

Пачечный upsert

POST /nexus/customer-db/api/tables/vk_clients/records/batch-upsert
Content-Type: application/json

{
  "records": [
    {
      "platform_id": "vk_123456",
      "custom_fields": {
        "name": "Иван Иванов",
        "source": "ВКонтакте"
      }
    }
  ]
}

Безопасность

Панель и чтение API доступны только авторизованным пользователям Nexus с доступом к модулю. Внешние запросы записи /records, /records/upsert и /records/batch-upsert принимают заголовок Authorization: Bearer <NEXUS_CUSTOMER_DB_API_TOKEN>. Для webhook-блоков без заголовков endpoint /records/upsert и короткий endpoint /{table} также принимают верхнеуровневое поле JSON/form token, api_token или secret; эти поля не сохраняются в карточку. Если отдельный токен не задан, используется NEXUS_OPENROUTER_API_TOKEN.

Текущий токен и готовый JSON заголовков доступны в верхней панели Nexus: откройте модуль База клиентов и нажмите Настройки рядом с кнопкой паузы.
